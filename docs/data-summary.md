# Data Summary: Belt Break Detection Dataset

Описание данных для задачи разметки. Источник: `Some data/Belt/`.

---

## 1. Задача и целевая метка

**Задача:** бинарная классификация — есть ли обрыв приводного ремня станка-качалки (СК) в данном временном окне.

| Метка | Значение | Описание |
|-------|----------|----------|
| `obryv = 0` | Норма | Штатная работа, нет признаков обрыва |
| `obryv = 1` | Обрыв ремня | Зафиксирован или предсказан обрыв приводного ремня СК |

**Замечание:** В `Some data/` на данный момент присутствует только один тип отказа — обрыв ремня. Остальные классы из исходного дизайна системы (`rod_break`, `idle_motor`, `overload` и т.д.) данными не подкреплены.

---

## 2. Физический процесс и оборудование

**Объект:** нефтяная скважина с механизированным способом добычи — **станок-качалка (СК)**, `prod_method_id = 'SE0012'`.

**Сигнал:** **активная мощность электродвигателя СК** (`tech_name = 'active_power'`, параметр `p18`), кВт. Измеряется SCADA-системой, интервал съёма ~30 сек.

- При нормальной работе СК потребляет мощность с характерной периодической формой — циклическое изменение с частотой качаний.
- При **обрыве ремня** двигатель перестаёт нагружать штанговую колонну: мощность резко падает, форма цикла искажается или становится плоской/хаотичной.
- Ложные срабатывания — плановые остановки, пуски, смена ремня, засорение клапанов — дают схожий сигнал.

**Геолокация скважин:** формат `m_geo` — `"МесторождениеПл_НомерСкважины"`, например `Абдуллинская_2248`, `Аксеновская_225ТТШ`, `Туймазинская_3301`. Месторождения в Башкортостане (Бавлинское нефтяное дивизион, БНД).

---

## 3. Источники данных и форматы

### 3.1 Временной ряд активной мощности — основной сигнал

Общая схема полей во всех вариантах датасета:

| Поле | Псевдонимы | Тип | Описание |
|------|-----------|-----|----------|
| Идентификатор скважины | `object_id`, `skv`, `m_geo` | int / str | В `combined_data.csv` — строка формата `"Поле_НомерСкв"` |
| Временная метка | `d`, `t`, `valuedate` | datetime | Интервал ~30 сек |
| Активная мощность | `p`, `v`, `p18`, `P` | float, кВт | Основной сигнал. `p18` — номер параметра в SCADA |
| Реактивная мощность | `p19` | float | Присутствует в `combined_data.csv`; значение `-32000` означает отсутствие данных (NULL) |
| Метка обрыва | `obryv` | 0 / 1 | Есть в датасетах для обучения; в production CSV — предсказание модели |

Фильтры качества: значения `P` вне `[-100, 100]` кВт отбрасываются. Окна с двумя подряд нулями (`find_zero`) пропускаются — признак останова скважины.

### 3.2 `combined_data.csv` — самый большой сырой датасет

**Путь:** `Обрыв ремней/Активная мощность ЭД СК/combined_data.csv`

| Параметр | Значение |
|----------|----------|
| Строк | ~10.1 млн |
| Уникальных скважин (`m_geo`) | 3 308 |

| Колонки | `valuedate`, `p18` (кВт), `p19` (реактивная или NULL=-32000), `m_geo`, `time_diff` (сек между замерами) |
| Интервал съёма | ~30 сек (`time_diff` ≈ 29–31 сек) |
| Метки `obryv` | **Отсутствуют** — это сырые данные без разметки |

Это основной кандидат для разметки агентской системой.

### 3.3 `_SELECT_202507231724.csv` и `_SELECT_202508150908.csv` — production inference

**Путь:** `Belt/`

Это результат работы продакшен-пайплайна: каждая строка — одно 3-часовое скользящее окно одной скважины с уже вычисленными признаками и предсказанием модели.

| Поле | Описание |
|------|----------|
| `object_id` | ID скважины (integer) |
| `well_name`, `area_name`, `field_name`, `shop_name` | Иерархия расположения (поле → куст → ГЗУ → ЦДНГ) |
| `values` | Raw-массив значений мощности в окне: PostgreSQL `{v1,...,vN}`, до 46 точек, ~3 ч |
| `window_start` / `window_stop` | Временные границы окна (~3 часа) |
| `std_day` | Стандартное отклонение за сутки по скважине |
| `model_input` | Вектор из 12 статистических признаков: `{mean, std, var, median} × {full, first_half, second_half}` |
| `point_count` | Число замеров в окне (1–46) |
| `obryv` | **Предсказание модели** (0 или 1), не ground truth |

**Размер и баланс:**

| Файл | Строк | `obryv=1` | Доля |
|------|-------|-----------|------|
| `_SELECT_202507231724.csv` | ~914 тыс. | 847 | **0.09%** |
| `_SELECT_202508150908.csv` | ~1.19 млн | — | — |

Дисбаланс продакшен-данных очень высокий — типичное поведение при низком пороге модели и преобладании нормальной работы.

### 3.5 XML / PNG примеры (131 файл)

`Belt/examples/file_1.xml` ... `file_131.xml` — SpreadsheetML-файлы, каждый содержит временной ряд активной мощности по одной скважине за **одни или несколько суток**, в течение которых произошёл обрыв ремня. Оригинальные имена файлов (до переименования) содержали: `"МесторождениеПл. ГЗУ ГXXXX Скв. YYYY Активная мощность.xml"`. Маппинг хранится в `Renamed *.txt`.

`Belt/examples images/file_*.png` — визуализации этих временных рядов.

**Важная особенность разметки:** по каждому примеру известна дата и приблизительное время отказа из отчётов АДКУ/ВСП, однако это зафиксированное время может расходиться с фактическим моментом обрыва на **несколько часов** — оборудование не всегда останавливается немедленно, а оператор может зафиксировать событие с задержкой. Это означает, что точная локализация момента отказа внутри суточного ряда является частью задачи разметки.

## 4. Признаковое пространство

### CatBoost-подход (финальная модель)

**Скользящее окно:** 2 часа, шаг 5–15 минут. Нулевые значения фильтруются.

12 признаков из окна (feature importance из `Belt_final.ipynb`):

| Признак | Описание | Importance |
|---------|----------|-----------|
| `median_first_half` | Медиана первой половины | 13.7% |
| `var_second_half` | Дисперсия второй половины | 12.2% |
| `std_second_half` | Стд второй половины | 11.7% |
| `var_first_half` | Дисперсия первой половины | 10.1% |
| `mean_second_half` | Среднее второй половины | 9.4% |
| `median_second_half` | Медиана второй половины | 8.8% |
| `std_first_half` | Стд первой половины | 7.7% |
| `median` | Медиана всего окна | 8.0% |
| `mean_first_half` | Среднее первой половины | 6.4% |
| `mean` | Среднее | 4.9% |
| `std` | Стд | 4.0% |
| `var` | Дисперсия | 3.0% |

**Ключевой паттерн:** расхождение статистик первой и второй половин окна. Момент обрыва ремня разбивает окно на "до" и "после" — медиана и дисперсия второй половины резко отличаются.

### Альтернативный подход — скользящее окно 3 ч (продакшен)

Из 3-часового скользящего окна вычисляются те же 12 статистических признаков и подаются в модель для онлайн-детекции.

## 5. Существующие модели

| Модель | Файл | Метод | Вход | Порог |
|--------|------|-------|------|-------|
| **CatBoost** (финальная) | `Final/catboost_model.cbm`, `.pkl` | Табличная классификация | 12 статистических признаков | настраиваемый |

**CatBoost обучение:**
- Optuna HPO: 50 trials, StratifiedKFold-5, метрика F1
- 1000 итераций, Logloss. Финальный train logloss ≈ 0.00284 (возможно переобучение на тренировочной выборке)
- Данные дополнялись ручной балансировкой (`additional_obryv`, `additional_norm`)
- Инференс: скользящее окно 2–3 ч, шаг 15 мин, визуализация результатов по `m_geo` через Plotly

## 6. Продакшен-пайплайн (`belt_control.py`)

```
1. Запрос к PostgreSQL (ts.ts.timeseries):
   - последние N минут активной мощности (prod_method = SE0012, param = active_power)
   - исключить скважины: в плановом стопе (adku.v_last_sost) ИЛИ в stop/start-режиме (ts.v_well_start_stop)

2. Для каждой скважины:
   - взять последние N замеров, фильтр: P ∈ [-100, 100] кВт
   - если два подряд P=0 → пропустить (find_zero = True)
   - вычислить 12 статистических признаков по скользящему окну

3. CatBoost.predict(features) → pred
   Если pred ≥ порога → создать рекомендацию type_id=15 ("belt_broken_risk")

4. Логировать в calc_logs.belt_breakage: (object_id, calc_date, values[], ml_bad)
```

Результат логируется **всегда**, рекомендация создаётся только при превышении порога. В `results.recommendation_d` запись удаляется автоматически если в следующем цикле обрыв не подтверждается; через 35 мин переносится в основную таблицу `recommendation`.

## 7. Качество и ограничения данных

| Аспект | Описание |
|--------|----------|
| **Дисбаланс классов** | В продакшен-выборке — 0.09% обрывов; в обучающем датасете Velena — 6.7% (специально обогащён аномалиями) |
| **Нулевые значения** | Два подряд нуля = останов, окно исключается |
| **Переменная длина окна** | 1–46 точек; окна < 5 точек пропускаются |
| **Базовый уровень per-well** | Разные скважины работают на разных уровнях мощности — нужна per-well нормировка для детектора аномалий |
| **`p19` в combined_data** | Почти всегда -32000 (NULL реактивной мощности), использовать нельзя |
| **Сезонный дрейф** | Не обрабатывается в текущих моделях |
| **Верифицированные vs неверифицированные** | Отчёты АДКУ 2024 помечены как непроверенные; для ground truth использовать только "Перечень (проверенный)" |
| **Переименование XML** | Оригинальные имена файлов содержали поле + ГЗУ + номер скважины; после анонимизации → `file_N.xml`. Маппинг в `Renamed *.txt` (кодировка cp1251 повреждена) |

## 8. Выводы для реализации агентской системы

**Роль этого датасета в новой архитектуре:**

Эти данные покрывают **один реальный task/domain pack** — обрыв ремня СК по активной мощности. Это не ограничение всей системы, а **первый reference domain** для универсального агента. В универсальной архитектуре таксономия задаётся через `TaskSpec`, а не глобально для всех задач сразу. Поэтому belt-break кейс должен использоваться как:

- первая валидируемая постановка задачи
- первый domain adapter
- первый источник confirmed examples и baseline-логики
- тест на то, что core можно потом переиспользовать для других установок и типов отклонений

**Адаптации под реальные данные:**

1. **Сигнал — не вейвформограмма в традиционном смысле.** Это дискретный временной ряд с шагом ~30 сек, а не один цикл насоса. Единица анализа — скользящее **2-часовое окно**, не отдельный цикл.

2. **Baseline per-well обязателен.** В `combined_data.csv` 3 308 скважин с разными номинальными уровнями мощности. Глобальный порог не работает. Базовый профиль строить из нормальных окон той же скважины.

3. **Knowledge base для retrieval** уже существует: отчёты АДКУ + ВСП 2023–2025 — это и есть maintenance logs. По `m_geo` + дате можно точечно проверить, был ли зафиксированный обрыв плановым событием или реальным отказом.

4. **Модель как один из сигналов.** Предсказание CatBoost (`ml_bad`, `obryv`) использовать как входной признак в контексте агента (аналог аномального скора), а не как метку.

5. **Временная неточность разметки.** Дата и приблизительное время отказа из отчётов АДКУ/ВСП могут расходиться с фактическим моментом обрыва на несколько часов. Агент должен работать с суточным или многосуточным временным рядом и локализовывать момент отказа по форме сигнала, а не просто доверять зафиксированному времени.

6. **`combined_data.csv` (10.1 млн строк, 3 308 скважин, 2.5 месяца)** — основной кандидат для массовой разметки агентом в belt-break task. Меток в нём нет, ground truth нужно строить через джойн с историческими отчётами АДКУ/ВСП по `m_geo` + дате.

7. **Переносимость на другие задачи ограничена доменной адаптацией.** Универсальный core можно переиспользовать, но для другой насосной установки или другого типа отклонения потребуются новый `TaskSpec`, новый adapter, свои quality-rules, baseline-логика и context sources.
